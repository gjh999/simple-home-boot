# 디지털 정부서비스 UI/UX 가이드라인 (2025.08) — 요약본

> 원문: 행정안전부 「디지털 정부서비스 UI/UX 가이드라인(2025.08)」 (`krds-uiux-가이드라인(2025.08).md`, 약 17,200줄).
> 본 문서는 **프런트엔드 구현 관점**에서 핵심만 추린 실무 요약본이다. 정확한 기준·도식은 원문/공식 사이트(https://www.krds.go.kr)를 확인한다.

---

## 0. 한눈에

- **KRDS(Korea Design System)** = 행정안전부 **범정부 표준 디자인 시스템**. 디지털 정부서비스(웹·모바일웹/앱)의 일관성·접근성·사용성 확보가 목표.
- 구성 5단계: **원칙 → 스타일 → 컴포넌트 → 기본 패턴 → 서비스 패턴**. (원칙이 나머지를 포괄)
- 적용 대상: 행정·공공기관이 구축·운영하는 **모든 웹사이트·모바일 웹/앱**.
- 접근성 목표: **한국형 웹콘텐츠 접근성 지침(KWCAG) / WCAG 2.1 AA**.
- 서비스 패턴 사용성은 **필수(Do) → 권장(Better) → 우수(Best)** 3단계로 점진 적용. **필수부터 우선 준수**.

---

## 1. 디자인 원칙 (7)

1. **사용자 중심의 서비스** — 실제 사용자 과업·여정 기준으로 설계.
2. **모든 사용자를 포용하는 서비스** — 장애·고령 등 포함(접근성).
3. **공통된 경험 안에서 개별 특성을 고려** — 일관성 + 서비스별 유연성.
4. **빠르고 간단한 서비스** — 단계·인지 부하 최소화.
5. **쉽게 이해하고 사용** — 쉬운 언어·명확한 레이블.
6. **다양한 상황을 고려** — 디바이스·환경·맥락 대응(반응형).
7. **신뢰할 수 있는 서비스** — 일관된 시각 신뢰, 명확한 피드백.

## 2. 스타일

### 2-1. 색상(Color)
- 팔레트: **Primary / Secondary / Gray / System(정보·성공·경고·위험)** — 시스템에 정의된 **색상값 그대로** 사용.
  - 본 프로젝트 기준값: Primary `#256EF4`, Primary-hover `#0B50D0`, Gray 스케일(`--krds-gray-*`).
- **상호작용 상태**(hover/active/focus/disabled): 기본 색 기준 **증·감 규칙**으로 표현.
- **시스템 색상**은 의미 체계(정보/성공/경고/위험) 준수.
- 배경·테두리·텍스트 색을 **일관성 있게** 적용. 명도 대비 충족(아래 접근성).

### 2-2. 타이포그래피(Typography)
- 서체: **Pretendard GOV**(정부 상징) / 자체 상징은 고딕 계열 허용.
- **본문 기본 크기 17px 이상**(자체 상징 16px 이상).
- **행간(line-height) = 글자 크기의 1.5배 이상**.
- **굵기 2종 이상** 구분, 각 굵기에 **맞는 폰트 파일** 사용.
- **Title·Heading은 본문보다 굵거나 크게**, 서체 규칙 일관 적용.

### 2-3. 형태(Shape)
- 모서리 반경(radius)·테두리 두께를 토큰 기준으로 **일관 적용**.

### 2-4. 레이아웃(Layout)
- 그리드·여백(spacing) 토큰 기반 배치, 반응형 브레이크포인트 고려.

### 2-5. 아이콘 / 엘리베이션 / 고대비 모드
- 아이콘: 시스템 아이콘 규격·의미 일관. 의미 전달은 텍스트 병행.
- **엘리베이션(그림자)**: 깊이 단계 토큰 사용.
- **선명한 화면 모드(High contrast)** 지원 고려.

### 2-6. 디자인 토큰(Design token)
- 색·서체·간격·형태·엘리베이션을 **토큰(약 1,835개)**으로 코드화 → 디자이너·개발자 공통 언어.
- KRDS 토큰은 **root font-size 62.5%(=10px)** 기준으로 저작됨 → 프로젝트 root가 16px이면 컴포넌트가 1.6배 커지므로 **정규화 필요**(본 프로젝트는 `krds.css`에서 `!important` 정규화로 해결, root는 16px 유지).

---

## 3. 컴포넌트 (구현 시 자주 쓰는 것 중심)

**탐색**: 건너뛰기 링크(Skip link), 메인 메뉴(GNB), 브레드크럼, 사이드 메뉴, 콘텐츠 내 탐색, **페이지네이션**, 탭바.
**레이아웃·표현**: 구조화 목록, 긴급 공지, 달력, 디스클로저, 모달, **배지**, **아코디언**, 이미지, 캐러셀, **탭**.
**선택**: 라디오·체크박스·셀렉트·태그.
**액션**: **링크 / 버튼**(계층 primary·secondary·tertiary, 크기 xsmall~xlarge).
**입력**: 텍스트 입력·텍스트 영역·날짜 입력·파일 업로드.
**아이덴티티**: 공식 배너, 운영기관 식별자, **헤더 / 푸터**.

> 각 컴포넌트는 ① 유형·상태 정의 ② 사용성 ③ 접근성 ④ 상호작용 ⑤ 플랫폼 고려를 함께 규정. 구현 시 KRDS 네이티브 클래스(`krds-*`, `tbl`, `form-group` 등) 사용.

---

## 4. 기본 패턴 / 서비스 패턴

- **기본 패턴**: 개인 식별 정보 입력, **입력폼**, 첨부파일, 필터링·정렬.
- **서비스 패턴**(사용자 여정): 방문, **검색**, **로그인**, **신청**, 정책 정보 확인.
  - 각 단계별 사용성 가이드를 **필수→권장→우수**로 제시.

---

## 5. 접근성(필수 체크 포인트)

- **건너뛰기 링크(skip-nav)** 제공.
- 키보드 포커스 가시성(**`:focus-visible` 링**), 논리적 탭 순서.
- **명도 대비**(본문 텍스트/배경 4.5:1 이상 등) 충족.
- 폼: `label`–입력 연결, 오류 메시지 명확, 필수 표시.
- 표: `caption`·`scope` 부여, 의미 구조 유지.
- 색만으로 정보 전달 금지(텍스트·아이콘 병행).
