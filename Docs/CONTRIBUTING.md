## 기여할 수 있는 항목

다음 카테고리 중 하나에 해당하는 자료를 등재할 수 있습니다.

| 카테고리 | 설명 |
|---------|------|
| 실행환경 / 런타임 | 표준프레임워크 런타임 확장·대체 구현 |
| 공통 컴포넌트 / 확장 모듈 | 표준프레임워크에 끼워 쓸 수 있는 컴포넌트 |
| 템플릿 / 보일러플레이트 | 프로젝트 시작 골격 |
| 개발 도구 | CLI, IDE 플러그인, 빌드 도구 등 |
| 통합 / 연동 | 외부 시스템·서비스 연동 모듈 |
| 샘플 / 예제 프로젝트 | 학습·참고용 완성 프로젝트 |
| 학습 자료 | 블로그, 영상, 강의, 책 |

해당 카테고리가 없다면 PR 본문에 **새 카테고리 신설을 제안**해 주세요.

## 등재 기준

### 필수 기준

다음 조건을 **모두** 충족해야 합니다.

- [ ] **공개 저장소**에 코드 또는 자료가 공개되어 있을 것 (GitHub, GitLab 등)
- [ ] **명확한 오픈소스 라이선스**가 표기되어 있을 것
      (Apache-2.0, MIT, BSD, MPL-2.0 등 OSI 승인 라이선스 권장)
- [ ] **README가 작성**되어 있고 다음을 포함할 것
  - 프로젝트 목적
  - 설치·사용 방법
  - 표준프레임워크와의 관계 (어떤 버전과 호환되는지)
- [ ] **유지보수 활동**이 최근 1년 이내 확인될 것
      (커밋, 릴리스, 이슈 응답 중 하나 이상)
- [ ] **악성 코드, 백도어, 광고성 자료**가 아닐 것

### 권장 기준

필수는 아니지만 등재 심사에 가산점이 됩니다.

- 테스트 코드 또는 CI 설정
- 표준프레임워크 호환 버전 명시 (예: `egov 4.x`, `egov 5.x`)
- 예제 또는 데모 환경 제공
- 한국어 README 제공

### 등재 불가

- 비공개·유료 자료
- 표준프레임워크와 관련 없는 일반 라이브러리
- 단순 포크 또는 미러 저장소 (의미 있는 차별점이 없는 경우)
- 광고, 프로모션, 회사 소개 페이지
- 보안 취약점이 알려진 채 패치되지 않은 프로젝트

## 작성 규칙

### 항목 형식

다음 형식을 **정확히** 지켜 주세요.

```markdown
- [저장소명 또는 자료명](URL) - 한 줄 설명 (마침표로 끝).
```

**예시**
```markdown
- [honggildong/egov-sso-login](https://github.com/honggildong/egov-sso-login) - SAML 기반 SSO 연동 로그인 컴포넌트.
```

### 세부 규칙

- **저장소명**: GitHub 경로 형식(`소유자/저장소명`) 또는 프로젝트 공식명 사용
- **URL**: `https://` 로 시작하는 정식 URL (단축 URL 금지)
- **설명**:
  - **한 줄** (80자 이내 권장)
  - **마침표(`.`)로 종료**
  - 한국어 또는 영어 (한국어 권장)
  - 과장된 형용사 금지 (예: "최고의", "유일한", "혁신적인")
- **정렬**: 카테고리 내에서 **알파벳 순서** (한글은 가나다 순)
- **중복 등재 금지**: 같은 프로젝트는 한 카테고리에만 등재

### 카테고리 선택

본인 프로젝트의 성격에 가장 가까운 카테고리 하나를 선택하세요.
여러 카테고리에 해당한다고 판단되면 PR 본문에 사유와 함께 적어 주세요.

## PR 제출 절차

### 1. 저장소 포크

GitHub에서 본 저장소를 본인 계정으로 포크합니다.

### 2. 브랜치 생성

```bash
git checkout -b add-honggildong-egov-sso-login
```

브랜치명 규칙:
- 신규 등재: `add-<프로젝트명>`
- 수정: `update-<프로젝트명>`
- 삭제: `remove-<프로젝트명>`

### 3. README 수정

해당 카테고리에 알파벳/가나다 순서로 한 줄을 추가합니다.

### 4. 커밋

커밋 메시지 형식:

```
add: <저장소명> 등재

- 카테고리: <카테고리명>
- 라이선스: <라이선스명>
```

**예시**
```
add: honggildong/egov-sso-login 등재

- 카테고리: 공통 컴포넌트 / 확장 모듈
- 라이선스: Apache-2.0
```

### 5. PR 생성

PR 제목과 본문은 다음 템플릿을 사용해 주세요.

**제목**
```
[add] <저장소명>
```

**본문**
```markdown
## 등재 신청 프로젝트
- 이름: <저장소명>
- URL: <URL>
- 라이선스: <라이선스명>
- 카테고리: <카테고리명>

## 등재 기준 체크리스트
- [ ] 공개 저장소에 공개되어 있음
- [ ] 오픈소스 라이선스가 명시되어 있음
- [ ] README에 목적·사용법·표준프레임워크 관계가 기술됨
- [ ] 최근 1년 이내 유지보수 활동이 있음
- [ ] 작성 규칙(형식·정렬·설명 길이)을 준수함

## 프로젝트 소개 (선택)
이 프로젝트가 표준프레임워크 생태계에 어떤 가치를 더하는지 간단히 설명해 주세요.
```

## 리뷰 절차

1. **자동 검증**: PR 생성 시 링크 유효성, 마크다운 형식이 자동 검사됩니다.
2. **메인테이너 리뷰**: 등재 기준 충족 여부, 카테고리 적절성을 검토합니다.
3. **피드백 반영**: 수정 요청이 있으면 PR에 추가 커밋으로 반영해 주세요.
4. **머지**: 기준 충족 시 메인테이너가 머지합니다. (보통 영업일 기준 7일 이내)

## 수정 및 삭제 정책

### 직접 수정 요청

본인 프로젝트의 설명, URL, 카테고리를 변경하고 싶다면 동일한 PR 절차를 따르세요.

### 자동 삭제 대상

다음 경우 메인테이너가 등재를 삭제할 수 있습니다.

- **저장소 비공개 전환** 또는 삭제
- **24개월 이상** 커밋·이슈 응답 등 활동 없음 (사전 이슈로 안내 후 처리)
- **라이선스 위반** 또는 라이선스 표기 제거
- **악성 코드, 보안 취약점 미패치** 확인
- **커뮤니티 가이드라인 위반** 신고가 접수되어 사실로 확인된 경우

### 본인 요청 삭제

본인 프로젝트의 등재 삭제를 원하시면 이슈를 생성해 주세요. 별도 심사 없이 처리합니다.

## 행동 강령

본 저장소의 모든 기여자는 [Contributor Covenant](https://www.contributor-covenant.org/ko/version/2/1/code_of_conduct/) 행동 강령을 따릅니다.

- 모든 참여자를 존중하는 언어 사용
- 건설적인 피드백
- 차별·괴롭힘·인신공격 금지

위반 사례는 운영팀 이메일(egovframesupport@gmail.com) 또는 GitHub Issue로 신고해 주세요.

## 자주 묻는 질문

**Q. 회사에서 만든 사내 프로젝트를 공개해 등재해도 되나요?**
A. 가능합니다. 공개 저장소에 오픈소스 라이선스로 공개되어 있다면 등재 신청할 수 있습니다.

**Q. 표준프레임워크 코어 PR과는 어떻게 다른가요?**
A. 이 저장소는 **링크만 모아 정리**합니다. 표준프레임워크 코어에 기여하고 싶다면
[egovframework 메인 저장소](https://github.com/egovframework)에 직접 PR을 보내 주세요.

**Q. 등재되면 표준프레임워크 공식 인증인가요?**
A. **아닙니다.** 본 목록은 커뮤니티에서 모아 정리한 자료이며, 등재 자체가 공식 보증을 의미하지 않습니다.
사용자에게는 이 점이 README 상단에 명시되어 있습니다.

**Q. 카테고리에 없는 종류의 프로젝트입니다.**
A. PR 본문에 새 카테고리를 제안해 주세요. 메인테이너가 검토 후 신설 여부를 결정합니다.

**Q. 등재가 거절되는 사유는 무엇인가요?**
A. 가장 흔한 거절 사유는 다음과 같습니다.
- 라이선스 미표기
- README 부실 (표준프레임워크와의 관계 불명확)
- 유지보수 활동 부족
- 작성 규칙 미준수 (형식, 정렬, 설명 길이)

거절 시 메인테이너가 사유를 PR 코멘트로 안내합니다. 보완 후 재신청 가능합니다.
